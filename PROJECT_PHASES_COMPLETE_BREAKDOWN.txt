AGRIBOT COMPLETE PROJECT PHASE BREAKDOWN (FIRST TO LAST)
Version date: 2026-03-25
Scope: This document summarizes the full project lifecycle and current implementation state, including models, approaches, and design rationale.

====================================================================
PHASE 1 - PROBLEM FRAMING AND PRODUCT GOAL DEFINITION
====================================================================
What was done:
- Defined AgriBot as a production-grade, offline-first agricultural decision-support system for Bangladesh.
- Set the primary requirement: multimodal input (text, voice, image) with bilingual output (English + Bangla).
- Added mandatory trust constraints: grounded answers, citations, conservative refusal behavior.
- Established strict edge hardware limits (e.g., 4GB VRAM): GPU reserved strictly for LLM, CPU for all other models.

Approach used:
- Offline Agentic RAG architecture with strict evidence and verification flow.
- Strict Hardware-Isolated Vision Architecture (CPU-Bound CNN + Heuristics) instead of memory-heavy VLMs.
- Production orientation (API contracts, diagnostics, testability, deployment artifacts).

Why this approach:
- Rural/field usage has unstable connectivity, so cloud-first was rejected.
- Agricultural advice is safety-sensitive; hallucination must be reduced with evidence policy.
- Edge consumer hardware (like RTX 3050 4GB) cannot run VLMs and conversational LLMs simultaneously without OOM crashes.


====================================================================
PHASE 2 - DATA INGESTION AND PDF UNDERSTANDING PIPELINE
====================================================================
What was done:
- Built ingestion stack to parse agricultural PDF manuals and convert into retrievable structured text chunks.
- Added provenance-preserving metadata (source file, page references).

Models/tools used:
- marker-pdf (layout-aware extraction)
- pypdf fallback extraction
- pytesseract + PyMuPDF OCR fallback for scanned content

Approach used:
- Prefer Marker output for structure (headings, tables, reading order).
- Fallback chain for robustness: Marker -> OCR/basic extraction.
- Noise filtering: repeated header/footer detection, TOC/reference down-weighting.

Why this approach:
- Real agricultural manuals are noisy and layout-heavy; plain linear extraction degrades retrieval quality.
- Provenance is required for citation-backed outputs.


====================================================================
PHASE 3 - CHUNKING AND INDEX ARTIFACT GENERATION
====================================================================
What was done:
- Built chunking and index build process with saved artifacts in data/indexes.
- Persisted chunks, dense vectors, FAISS index, BM25 index, and manifest.

Models/tools used:
- SentenceTransformer embedding model: sentence-transformers/all-MiniLM-L6-v2
- FAISS (IndexFlatIP, cosine-equivalent with normalized embeddings)
- BM25 (rank-bm25)

Approach used:
- Hybrid indexing (dense + sparse) with persistent local artifacts.
- Keep-weight metadata to reduce low-quality/redundant page influence.

Why this approach:
- Dense search captures semantics, sparse search captures exact terms/numbers.
- Local persisted indexes guarantee offline operation and fast startup.


====================================================================
PHASE 4 - DIALECT KNOWLEDGE GRAPH (KG BEYOND A LEXICON)
====================================================================
What was done:
- Implemented SQLite-backed offline knowledge graph schema and seed data.
- Mapped dialect terms to canonical Bengali/English/scientific entities.
- Added structured relation links (symptom->disease, disease->treatment).

Models/tools used:
- SQLite graph tables (portable, versionable KG schema)
- Entity linker, disambiguation, and query expansion logic

Approach used:
- Query tokenization with unigram/bigram/trigram matching.
- Alias/canonical linking + neighbor expansion to enrich retrieval query before semantic search.

Why this approach:
- Farmers use colloquial/dialect terms that break standard dense/sparse vector retrieval against scientific manuals.
- Graph relationships provide material reasoning enhancements and high disambiguation quality for Agentic RAG.


====================================================================
PHASE 5 - HYBRID RETRIEVAL AND RERANKING LAYER
====================================================================
What was done:
- Implemented HybridRetriever and reranker stage.
- Combined dense and sparse retrieval using weighted reciprocal rank fusion.

Models/tools used:
- Dense retrieval: all-MiniLM-L6-v2 embeddings + FAISS
- Sparse retrieval: BM25
- Reranker: cross-encoder/ms-marco-MiniLM-L-12-v2 (loaded through reranker module)

Approach used:
- Retrieve candidate evidence from both channels.
- Fuse by weighted RRF.
- Apply cross-encoder reranking and threshold filtering.

Why this approach:
- Hybrid + rerank improves precision over single retriever systems.
- Especially important for technical agricultural terminology and dosage-sensitive context.


====================================================================
PHASE 6 - AGENTIC ORCHESTRATION WITH CONVERSATIONAL FALLBACK
====================================================================
What was done:
- Built LangGraph state-machine pipeline with conditional retry and conversational fallback behavior.

Models/tools used:
- LangGraph workflow in agent graph
- LLM-based grading/rewrite/verify/fallback helpers

Approach used:
- Pipeline: normalize -> kg_link -> vision heuristic fusion -> retrieve -> grade -> (rewrite/fallback) -> generate -> verify.
- Bounded retry count to avoid runaway loops.
- Conversational Fallback: Agent proactively pauses RAG loop to ask the farmer follow-up questions if visual context is lacking (e.g., severity).

Why this approach:
- Single-pass RAG fails on weak retrieval contexts.
- The lack of an intensive generalized VLM requires procedural intelligence to gather exact context.
- Bounded correction gives better groundedness while keeping latency and token costs scalable securely.


====================================================================
PHASE 7 - LOCAL LLM GENERATION ENGINE
====================================================================
What was done:
- Integrated local GGUF inference engine for answer generation and support tasks (grade/rewrite/verify prompts).

Models/tools used:
- Llama.cpp via llama-cpp-python
- Current configured model path default: models/qwen2.5-1.5b-instruct-q8_0.gguf

Approach used:
- Low temperature generation for factual stability.
- Evidence-constrained prompt templates with citation requirements.

Why this approach:
- Fully local inference is required for offline deployment.
- Smaller quantized model keeps memory/latency feasible on consumer hardware.


====================================================================
PHASE 8 - TRANSLATION LAYER FOR BILINGUAL DELIVERY
====================================================================
What was done:
- Added bidirectional translator module and Bangla normalization.

Models/tools used:
- csebuetnlp/banglat5_nmt_en_bn (EN -> BN)
- csebuetnlp/banglat5_nmt_bn_en (BN -> EN)
- bnunicodenormalizer

Approach used:
- Sentence-aware translation for better quality.
- Preserve citation text while translating answer body.

Why this approach:
- Retrieval/generation quality is strongest in English evidence format in many cases.
- End users need Bangla output with technical consistency.


====================================================================
PHASE 9 - VOICE INPUT/OUTPUT BASELINE (OFFLINE STT/TTS)
====================================================================
What was done:
- Implemented voice endpoints and audio preprocessing.
- Added offline text-to-speech output.

Models/tools used:
- STT baseline: faster-whisper (CTranslate2)
- TTS: pyttsx3 with system voices
- Audio preprocessing: ffmpeg-based canonicalization

Approach used:
- Voice request flow with confidence and warning diagnostics.
- Voice confirmation mechanism for uncertain transcript before running full RAG.

Why this approach:
- Voice is primary UX path for farmers.
- Confirmation step prevents low-confidence ASR from producing unsafe advice.


====================================================================
PHASE 10 - BANGLA ASR HARDENING AND MULTI-BACKEND FALLBACKS
====================================================================
What was done:
- Added robust Bangla-specific fallback architecture on top of Whisper.
- Integrated and stabilized BanglaSpeech2Text behavior in runtime.
- Added Vosk fallback path and readiness checks.
- Added script/repetition/quality heuristics to reduce transcript hallucination.

Models/tools used:
- Primary: faster-whisper (medium, beam=8, lang hint bn)
- Bangla fallback package: BanglaSpeech2Text==0.0.3 (official API path)
- Bangla fallback model (transformers fallback): bangla-speech-processing/BanglaASR
- Optional fallback: Vosk Bengali model at models/vosk-bn

Approach used:
- Try Whisper candidates first.
- Trigger fallback on low confidence, script mismatch, repetitive/no-speech warnings.
- If package model path fails in runtime, fallback to transformers pipeline.
- Use wave+numpy audio array path to avoid ffmpeg filename-decoding dependency in fallback ASR.
- Apply script-aware confidence calibration (short/Latin output penalties, mismatch warnings).

Why this approach:
- Real mic Bangla data quality varied, and single-backend ASR produced unstable transcripts.
- Multi-backend selection + warning-aware scoring gave better practical reliability.


====================================================================
PHASE 11 - CPU-BOUND VISION PIPELINE (MOBILENETV3 + HEURISTICS)
====================================================================
What was done:
- Implemented a hardware-isolated image endpoint using a deterministic, lightweight CNN and color heuristics.
- Integrated output schemas with the LangGraph conversational fallback logic.

Models/tools used:
- MobileNetV3 (ONNX Runtime, strictly localized to CPU)
- PIL + numpy color heuristics (symptom isolation, e.g., "yellowing")

Approach used:
- Completely offload visual diagnosis to CPU RAM.
- Feed Top-K disease classes + heuristic hints to the LangGraph Agent.
- Explicit agentic prompting ensures follow-ups requested from the user if accuracy implies risk.

Why this approach:
- General VLMs crash 4GB GPUs or hallucinate context arbitrarily.
- A deterministic CNN supported by robust heuristic rules enables fully concurrent multimodal request processing without OOM errors.

Models/tools used:

Why this approach:
- Production systems need observability and predictable contracts.
- Safety and debugging require per-request timings and warnings.

CURRENT MODEL/APPROACH SNAPSHOT (AS-IMPLEMENTED NOW)
====================================================================
Core LLM:
- qwen2.5-1.5b-instruct-q8_0.gguf via llama-cpp-python.

Embedding and retrieval:
- sentence-transformers/all-MiniLM-L6-v2
- FAISS + BM25 hybrid fusion
- Cross-encoder reranker with threshold filtering.

Translation:
- BanglaT5 EN<->BN pair from csebuetnlp + bn unicode normalization.

Voice ASR:
- Primary faster-whisper medium + quality heuristics.
- BanglaSpeech2Text package path plus transformers fallback (BanglaASR) when package model fetch path fails.
- Optional Vosk Bengali fallback if model files are present.

Voice TTS:
- pyttsx3 system voices (Bangla voice if available, else fallback).

Vision:
- MobileNetV3 ONNX + OpenCV/NumPy color heuristics (strictly CPU-bound).
- Agentic conversational fallback for handling low visual confidence.

Safety:
- Strict grounding mode support, verification checks, retry loop, refusal/caveat behavior.
- Voice confirmation gate for low-confidence transcript.


====================================================================
WHY THIS PHASED DESIGN WORKS FOR THIS PROJECT
====================================================================
- It is aligned with offline constraints from day one.
- It separates ingestion-time heavy work from query-time inference.
- It combines deterministic retrieval structure with bounded agentic correction.
- It introduces multilingual and speech support without sacrificing traceability.
- It keeps safety and diagnostics visible in API responses for both developers and operators.
